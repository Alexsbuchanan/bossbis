# BossBiS — Design Spec

**Date:** 2026-06-05
**Revision:** 2 (2026-06-05) — substantially expanded after a multi-agent audit of the weirdgloop calc internals, the RuneLite integration surface, and an adversarial review of revision 1. Every load-bearing claim in this revision is backed by a `file:line` citation into the pinned upstream (`/tmp/osrs-dps-calc-audit`, weirdgloop@`5e313c1`) or the local RuneLite checkout.
**Status:** Draft, pending user review.

> **Revision-2 note on what changed and why.** Revision 1 contained several factual errors that an audit surfaced: it claimed weirdgloop ships no tests (it ships 26 Jest files including a generated-oracle suite); it modeled player boosts as a potion name (the engine consumes a per-skill integer delta); it ignored `Monster.inputs` (raid scaling + the boss's own protection prayers, which can zero a style's damage); it assumed `src/lib` was self-contained (it depends on d3-array and lodash); it stated a `1e-12`/bitwise float-parity tolerance (impossible given an order-dependent, epsilon-truncated TTK fold); it left the optimizer's greedy/set-bonus composition unsound; and it assumed a two-repo + JitPack distribution that the RuneLite Plugin Hub cannot build. This revision corrects all of these. Section §0 is a changelog from revision 1.

## 1. Goals & non-goals

### Goals

- **Headline (v0.2):** Open the bank, pick a boss, and see the best loadout assemblable from owned items, with explicit "you're missing X" callouts and required-consumable checks.
- **v0.1 milestone:** A faithful Java port of the OSRS Wiki's open-source DPS calculator (`weirdgloop/osrs-dps-calc`), exposed via a RuneLite plugin panel that mirrors weirdgloop's input UI so we can verify parity end-to-end.
- **Free and open-source for the community.** Distributed via the RuneLite Plugin Hub.

### Non-goals

- **Spec-attack *chaining* in the optimizer** (model "fire a spec, then continue the fight" as a DPS-ranking input). The calc's spec code (`getSpecCalc()`) **is** ported faithfully — it is interleaved in the main path and the parity corpus exercises it — but the v0.2 optimizer ranks loadouts by sustained `getDps()`, not by spec damage. See §1.1 for why this distinction matters.
- **Within-kill prayer/style micro-optimization** (switching prayers/styles mid-fight to shave ticks). Note this is *distinct* from planner-level style/spell *selection*, which the optimizer **does** perform (§8), and from a boss's *own* protection prayers, which are an encounter fact the calc **does** model (§4, §8).
- **Team raids (CoX/ToB/ToA) as coordinated encounters.** The calc *can* score a single player against a raid-scaled monster (the scaling inputs are ported), but the v0.2 optimizer ships curated **solo** encounters only. Raid-scaled encounters are gated behind explicit per-encounter inputs (§8) and deferred to v0.3+.
- **Online stat tracking / account databases.** Local only; the plugin makes **zero** network requests (§7.6).
- **Inventory packing optimization** (food-per-minute, brew thresholds). Beyond flagging *required* consumables (antifire, Crumble-Undead runes, etc.), trip composition is out of scope; v0.3 stretch at most.

### 1.1 Why "port the spec code but don't optimize on it"

`PlayerVsNPCCalc.getSpecCalc()` (`PlayerVsNPCCalc.ts:2694`) rebuilds the calc with `usingSpecialAttack=true` and is invoked inside the worker's basic compute path (`worker.ts:19-61`). A 1:1 port that excised spec logic would diverge from upstream on the very rows the corpus checks, so **the code is ported**. What is *out of scope* is the optimizer reasoning about spec rotations (e.g. "bring a DWH to lower defence, then switch"). The optimizer ranks by `getDps()` only. This resolves the revision-1 contradiction where the non-goals excluded spec attacks but `PORT_STATE.md` listed them as "pending."

## 2. License & attribution

### 2.1 Why GPLv3

Porting weirdgloop's TypeScript (GPLv3, `LICENSE` at upstream root) makes our calc a derivative work, so **the plugin source is GPLv3**. This is not a free choice — it is mandated the moment we copy/port their code. (The item/monster *data* is separately wiki-derived and carries CC-BY-SA; see §2.3.)

### 2.2 Plugin Hub license risk (flagged)

GPLv3 acceptance on the Plugin Hub is **not documented and is unconventional** (the hub README's "Licensing your repository" section instructs selecting *BSD 2-Clause*; nearly all hub plugins are BSD-2-Clause). However, the hub's **automated** gate is more permissive than the README: the packager's `realPluginChecks` requires only that a `LICENSE` file exist at the repo root and permit RuneLite "to freely distribute the plugin jar standalone" (`plugin-hub-tooling` `packager/.../Plugin.java:982-994`), with BSD-2-Clause *recommended, not required*. GPLv3 permits binary redistribution (with source/offer), so it should pass the automated check, but **copyleft on the hub is a judgment call for the human reviewer.**

**Action:** Before submission, confirm GPLv3 acceptance with RuneLite maintainers (Discord / a draft PR). If they decline, the fallback is a **clean-room reimplementation** of the formulas (no copied code) under BSD-2-Clause — significantly more work and out of scope for this spec, but the escape hatch exists. This is the single biggest external risk to distribution (§11) and is surfaced as an open decision (§12).

### 2.3 Attribution (required, in README + plugin panel "About" + source headers)

> BossBiS ports damage formulas from the OSRS Wiki's open-source DPS calculator (github.com/weirdgloop/osrs-dps-calc), authored by Jayden Bailey / WeirdGloop, used under GPLv3. Item and monster data are derived from the OSRS Wiki under CC-BY-SA. Bundled data snapshot: weirdgloop@`<commit>`, `<date>`.

## 3. Repository layout & distribution

### 3.1 The Plugin Hub constraint that drives this section

The Plugin Hub does **not** host source. The plugin's code lives in our own public GitHub repo; we submit a 2-line manifest to a fork of `runelite/plugin-hub`. Crucially, the hub's packager runs Gradle at the **repo root** and bundles **only that root project's** `sourceSets.main.output` + `runtimeClasspath` into a single `plugin.jar` (`plugin-hub-tooling` `target_init.gradle`, task `runelitePluginHubPackage`). Therefore:

- A `:calc` + `:plugin` two-module layout where the hub is pointed at the plugin module **will not pull in `:calc`** unless `:calc` is on the plugin's `runtimeClasspath` and within the same build.
- **No JitPack / no separately-published artifact.** Any non-`runelite-client`-transitive dependency must be added to the hub's `verification-template/build.gradle` `thirdParty` allowlist with a recorded SHA-256 and **manually approved** by a maintainer — a slow path the README tells you to avoid. A separately-published calc artifact would trip this.

### 3.2 Chosen layout: single repo, calc vendored as internal source

```
bossbis/                              ← one public GitHub repo, GPLv3
├── settings.gradle                  ← single root project (Groovy DSL, mirroring runelite/example-plugin)
├── build.gradle                     ← Groovy DSL; build = "gradle" (§3.4); compileOnly net.runelite:client
├── runelite-plugin.properties        ← hub metadata (displayName, author, description, tags, plugins=, version, build)
├── icon.png                          ← ≤ 48×72 px (the only hub-enforced size limit)
├── LICENSE                           ← GPLv3, at repo root (hub requires a redistributable LICENSE)
├── PORT_STATE.md                     ← parity-tracking table (Java ↔ TS), per upstream commit
├── README.md                         ← attribution front-and-center
├── src/main/java/org/bossbis/
│   ├── calc/                         ← VENDORED Java port of weirdgloop src/lib (+ runtime support)
│   ├── plugin/                       ← RuneLite plugin (Plugin, Config, panel, scanner, optimizer)
│   └── ...
├── src/main/resources/org/bossbis/data/
│   ├── equipment.json                ← COMMITTED snapshot (not a build-time submodule copy)
│   ├── monsters.json                 ← COMMITTED snapshot
│   ├── equipment_aliases.json
│   ├── spells.json
│   └── data-version.properties       ← weirdgloop commit + snapshot date, read at runtime (§7.5)
├── src/test/java/org/bossbis/        ← parity + unit + optimizer-oracle tests
├── src/test/resources/parity/        ← committed corpus + ported upstream Jest scenarios
└── tools/                            ← DEV-ONLY, not built into the jar
    ├── upstream/                     ← git submodule: weirdgloop/osrs-dps-calc (porting reference + corpus gen)
    ├── gen-reference-corpus.ts       ← Node; imports tools/upstream/src/lib + utils + enums
    └── update-data.sh                ← refresh the committed JSON snapshot from the submodule
```

Key differences from revision 1:

- **The calc is vendored as internal `org.bossbis.calc.*` source**, not a separate Gradle module on the hub path. (For *local development* we may still use a Gradle subproject; but the **submitted** repo's root build must compile the calc into the one jar. The simplest way that guarantees this is a single module with the calc as packages — chosen here.)
- **The JSON data is committed into `src/main/resources`** as a snapshot. The build does **not** depend on the submodule (revision 1 copied data from a submodule via `processResources`, which would force the hub/CI to init submodules — a known gotcha). Loading uses `Class.getResourceAsStream` exclusively (the jar is never unpacked on the hub).
- **The weirdgloop submodule is dev-only**, under `tools/upstream/`, used to read the TS while porting and to run the corpus generator. It is `.gitignore`-excluded from the jar and irrelevant to the hub build.

### 3.3 Submission & update mechanics

1. Push BossBiS to a public GitHub repo with `runelite-plugin.properties` at root: `displayName=BossBiS`, `author=<you>`, `description=<…>`, `tags=boss,dps,gear,bis`, `plugins=org.bossbis.plugin.BossBisPlugin`, `version=<optional>`, `build=gradle`.
2. Fork `runelite/plugin-hub`, add `plugins/bossbis` containing exactly:
   ```
   repository=https://github.com/<you>/bossbis.git
   commit=<full 40-char SHA>
   ```
3. Open a PR. Hub CI clones at that commit, builds, and runs automated checks (build success, Java 11 bytecode, no `net.runelite.*` plugin classes, no blocklisted APIs, LICENSE present, valid `plugins=` classes, jar size). A human then reviews for malicious behaviour and Jagex third-party-client compliance.
4. **Updates** = a follow-up PR bumping only the `commit=` line.

### 3.4 Build constraints to honor (from the packager source)

- **Java 11 bytecode** for all plugin classes.
- Plugin classes must **not** live under `net.runelite.*`.
- **Blocklisted APIs are fatal**: no `new OkHttpClient()`, **no `new Gson()`** (inject the client's `Gson` instance), no `Client.getVar`, no `WidgetInfo`. This directly affects the data loader: `EquipmentRepository`/`MonsterRepository` take an **injected** `Gson`. **The client's bundled Gson is 2.8.5** (verified by building against `net.runelite:client:1.12.28`), so the data loader must use the 2.8.5 API surface — e.g. `new JsonParser().parse(Reader)` / `Gson.fromJson`, **not** the static `JsonParser.parseReader` (added in Gson 2.8.6).
- `build=standard` would let the hub replace our `build.gradle` with its template (faster review) but only if we are single-module with **zero** custom build steps and no non-RL deps. We are dependency-free (Gson is a `runelite-client` transitive, injected), so `build=standard` is *possible*; however the corpus/oracle test wiring and `strictfp`/test config are cleaner under `build=gradle`. **Decision: `build=gradle`**, with tab-indented, ≤120-col Gradle files (a hub lint).
- **Default 10 MiB jar limit.** Our committed JSON is ~5.4 MiB (equipment 3.1 + monsters 2.4); compiled classes are small, and there are no bundled third-party jars, so we fit with headroom (warnings start at 8 MiB). If a future data bump pushes us over, add `jarSizeLimitMiB=N` to the manifest (needs maintainer approval). See §7.7 for the heap-vs-jar distinction.

### 3.5 Tracking upstream weirdgloop releases

- Submodule `tools/upstream/` pinned to a commit; also add weirdgloop as a plain git remote so `git log weirdgloop/main` is queryable without polluting our history.
- **Data refresh** = `tools/update-data.sh <commit>`: checks out the submodule at `<commit>`, copies `cdn/json/*.json` into `src/main/resources/org/bossbis/data/`, writes `data-version.properties`, and we commit the diff. The diff is the changelog of what weirdgloop changed.
- **Code refresh** = re-read the changed TS in the submodule, update the corresponding Java, regenerate the corpus (§5), update `PORT_STATE.md`. Commit convention: `port: BaseCalc <weirdgloop@5e313c1>`, `sync: weirdgloop -> <commit>`.

### 4.2 Type model (exact, from `src/types`)

All immutable. Use **Lombok `@Value`** (JDK 11; records avoided to keep one toolchain). Field-by-field, verified against upstream:

- `PlayerSkills(int atk, def, hp, magic, prayer, ranged, str, mining, herblore)` — `Player.ts:8-18`. **`mining` and `herblore` are required** (some boosts/effects use them). `Player.boosts` is *also* a `PlayerSkills` holding **additive deltas** (current = base + boost; may be negative) — `Player.ts:117`.
- `PlayerBonuses(int str, rangedStr, magicStr, prayer)`; `PlayerOffensive`/`PlayerDefensive(int stab, slash, crush, magic, ranged)` — `Player.ts:54-75`.
- `EquipmentStats(PlayerBonuses bonuses, PlayerOffensive offensive, PlayerDefensive defensive)`.
- `EquipmentPiece extends EquipmentStats { String name; int id; double weight; String version; Slot slot; String image; int speed; EquipmentCategory category; boolean isTwoHanded; ItemVars itemVars; }` — `Player.ts:20-52`. `ItemVars(String blowpipeDartName, Integer blowpipeDartId)`.
- `PlayerEquipment` = fixed 11 nullable slots: head, cape, neck, ammo, weapon, body, shield, legs, hands, feet, ring — `Player.ts:20-52`.
- `Player extends EquipmentStats { String name; PlayerCombatStyle style; PlayerSkills skills; PlayerSkills boosts; PlayerEquipment equipment; int attackSpeed; List<Prayer> prayers; Buffs buffs; Spell spell /*nullable*/; }` — `Player.ts:77-174`. Player carries **both** the aggregated `EquipmentStats` and the raw `equipment` map.
- `Buffs(boolean onSlayerTask, inWilderness, forinthrySurge; int soulreaperStacks, baAttackerLevel, chinchompaDistance; boolean kandarinDiary, chargeSpell, markOfDarknessSpell, usingSunfireRunes)` — `Player.ts:121-168`. `buffs.potions` exists upstream but is **UI-only** ("should NOT be used in calculator code" — `Player.ts:121-128`); we omit it from the calc model.
- `PlayerCombatStyle(String name, CombatStyleType type, CombatStyleStance stance)`; `CombatStyleType ∈ {STAB,SLASH,CRUSH,MAGIC,RANGED, null}` — `PlayerCombatStyle.ts:4-64`.
- `Spell(String name, image; int maxHit; Spellbook spellbook; Spellement element; boolean unselectable)` — `Spell.ts:3-12`.
- `Monster(int id; String name, image, version; int size, speed; CombatStyleType style; MonsterSkills skills; MonsterOffensive offensive; MonsterDefensive defensive; Set<MonsterAttribute> attributes; Weakness weakness /*nullable*/; Immunities immunities; boolean isSlayerMonster; MonsterInputs inputs)` — `Monster.ts:15-58`. `defensive` carries `flatArmour` plus `{stab,slash,crush,magic}` and the **three ranged subtypes** `{light,standard,heavy}`. `maxHit` is UI-only — excluded from calc logic.
- **`MonsterInputs`** (`Monster.ts:63-132`) — the previously-missing encounter block: `boolean isFromCoxCm; int toaInvocationLevel, toaPathLevel, partyMaxCombatLevel, partySumMiningLevel, partyMaxHpLevel, partySize, monsterCurrentHp; DefenceReductions defenceReductions; Integer demonbaneVulnerability; String phase; MonsterPrayers prayers(boolean melee, ranged, magic)`. `DefenceReductions(boolean vulnerability, accursed; int elderMaul, dwh, arclight, emberlight, bgs, tonalztic, seercull, ayak)`.
- **Defaults** = `INITIAL_MONSTER_INPUTS` (`Monsters.ts:32-58`): `partyMaxCombatLevel=126, partySumMiningLevel=99, partyMaxHpLevel=99, partySize=1, toaInvocationLevel=0`, all reductions/prayers `false/0`, and `monsterCurrentHp` **reset to `monster.skills.hp` on monster select** (`MonsterContainer.tsx:148-149` — the Java port must replicate this reset).

### 4.3 Enums

- `Prayer` — explicit ordinals 0..30 matching `Prayer.ts:35-67` (`BURST_OF_STRENGTH=0 … PRESERVE`). Each carries `PrayerData(int renderOrder; String name; double drainRate; CombatStyleType combatStyle; Double magicDamageBonus; Factor factorAccuracy, factorStrength, factorDefence, factorDefenceMagic)`. **`Factor` = `int[2] {numerator, denominator}`** (denominator usually 100) so additive prayers combine correctly (`Prayer.ts:93-112,227-236`).
- `Potion` (22 members, `Potion.ts:1-26`), `MonsterAttribute` (16 string-valued, incl. `vampyre1/2/3`, `Monster.ts`/`MonsterAttribute.ts`), `EquipmentCategory` (string-valued, ~30, `EquipmentCategory.ts:5-46`), `BurnImmunity{WEAK,NORMAL,STRONG}`, `UserIssueType` (12 members, `UserIssueType.ts`), `FeatureStatus`.

### 4.4 Ported runtime support (the non-`src/lib` dependencies)

`src/lib` is **not** self-contained. The 1:1 port must also re-implement, with matching semantics:

| Upstream symbol | Used in (`src/lib`) | Java equivalent + semantic notes |
|---|---|---|
| d3-array `sum(iter, acc?)` | HitDist, Equipment, NPCVsPlayerCalc, PlayerVsNPCCalc, dists/claws | **Left-to-right** accumulation `acc += +v`; **skips values coercing to falsy** (NaN/undefined/null/0); returns 0 for empty. Verified against d3 source — it is **plain sequential, not Kahan-compensated**. Port must iterate in upstream order and skip non-finite/zero; **never** a parallel/reordered stream. |
| d3-array `max(iter, acc?)` | Comparator, scaling/ChambersOfXeric | Returns `undefined` for empty; callers rely on `?? 1` / `|| 0` (e.g. `ChambersOfXeric.ts:48-49`). Java returns `OptionalDouble`/sentinel; preserve fallbacks. |
| d3-array `some` / `range` / `min` / `cross` | HitDist, dists/bolts, PlayerVsNPCCalc | `some`=short-circuit existential; `range`=half-open int range (TTK iteration); `cross`=cartesian product (distribution zip). |
| `lodash.mergewith` | `Comparator.ts:123`, and via `@/utils typedMerge = merge({},base,updates)` (`utils.ts:164`) | Recursive deep merge. **No custom customizer is passed** at either call site, so default `mergeWith` array/null handling applies (source arrays overwrite by index; `undefined` sources skipped). A non-trivial deep-merge of immutable trees — verify array-index + undefined-skip behaviour. |
| `@/utils` (`keys, isDefined, getCombatStylesForCategory, typedMerge, FeatureStatus`), `@/enums/*`, `type-fest PartialDeep` | BaseCalc, Equipment, Comparator, PlayerVsNPCCalc, DefenceReduction | Ported as `support/Utils.java` + the enums above. The **corpus generator** (`tools/gen-reference-corpus.ts`) also needs these — so the dev submodule scope must include `src/utils.ts`, `src/state.tsx`, `src/enums`, `src/types`, plus `node_modules` (d3-array, lodash.mergewith). |

A dedicated `RuntimeSupportParityTest` exercises `D3.sum/max`, `MergeWith.merge`, and `Utils.typedMerge` against d3/lodash output for edge inputs (empty, NaN, undefined, mixed-magnitude ordering, nested arrays).

### 4.5 Public API of `org.bossbis.calc`

Small surface; plugin code never imports `BaseCalc`/`HitDist` directly.

```java
public final class DpsCalc {
  public static DpsCalc fromBundledData(Gson gson);                 // loads committed resources (injected Gson)
  public static DpsCalc fromRepositories(EquipmentRepository, MonsterRepository, SpellRepository);

  public CalcResult forPlayerVsNpc(Player player, Monster monster);
  public CalcResult forPlayerVsNpc(Player player, Monster monster, CalcOpts opts);
  public NpcVsPlayerResult forNpcVsPlayer(Monster monster, Player player);

  public EquipmentRepository equipment();
  public MonsterRepository monsters();
  public SpellRepository spells();
}
```

`CalcResult` mirrors upstream `PlayerVsNPCCalculatedLoadout` (`State.ts:48-85`) — all fields nullable:

```java
@Value public class CalcResult {
  Integer npcDefRoll, maxHit, maxAttackRoll;
  Double expectedHit, accuracy, dps, ttk;
  AttackDistribution distribution;           // exposes asHistogram(), getMax(), getExpectedDamage()
  Map<Integer,Double> ttkDist;
  Integer specMaxHit; Double specAccuracy, specExpected, specMomentDps, specFullDps;
  List<UserIssue> userIssues;
  List<DetailEntry> details, specDetails;    // only when CalcOpts.detailedOutput
}
```

- **`accuracy` vs display:** upstream distinguishes `getHitChance()` from `getDisplayHitChance()` (the latter blends Brimstone-ring magic 0.75/0.25; the worker uses *display*). The corpus asserts **both** where they differ; `CalcResult.accuracy` carries `getHitChance()` and the UI calls `getDisplayHitChance()` for presentation. (Resolves the revision-1 ambiguity.)
- **`CalcOpts`** (`BaseCalc.ts:23-59`): `loadoutName?, detailedOutput?, disableMonsterScaling?, usingSpecialAttack?, isBlindBag?, blindBagDistance?, blindBagUniques?, isEcho?, overrides?{accuracy?,attackRoll?,defenceRoll?,maxHit?:int[2]}`.

Constants `DEFAULT_ATTACK_SPEED=4`, `SECONDS_PER_TICK=0.6`.

### 4.6 Equipment aggregation (`Equipment.calculateEquipmentBonusesFromGear`)

`Equipment.ts` — (1) canonicalize all 11 slots via the alias map (§6); (2) sum each piece's `bonuses/offensive/defensive`, **skipping** ammo-slot `rangedStr`/`offensive.ranged` unless `ammoApplicability(weaponId, ammoId)==INCLUDED`; (3) apply ordered special cases (blowpipe dart `rangedStr`, Tumeken's Shadow magic ×3/×4 cap 1000, Dinh's bulwark str-from-defence, Virtus, Void mage 5%, Dizana's quiver, Keris partisan penalty, Crystal blessing); (4) set `attackSpeed = calculateAttackSpeed(player, monster)` (`Equipment.ts:249-313`: weapon speed → Rapid −1 → cast stances force 5/4/6 → Scurrius bone →1 → leagues → `max(result,1)`). Mirror the `ammoForRangedWeapons` table and `equipmentAliases` exactly.

### 4.7 Integer-math discipline & determinism (port rules)

Determinism porting rules — promoted from a footnote to a hard contract because parity depends on it:

- **No RNG anywhere** in `src/lib` (verified: zero `Math.random`/`crypto`/`Random`). Add a **CI grep guard** that fails if any RNG token appears in `org.bossbis.calc`.
- **All intermediate stat/roll/damage arithmetic is truncating** (`Math.trunc`, `Math.ts:11-16`). Java helpers: `iSqrt(x)=trunc(sqrt(x))`, `iLerp`, `addPercent(x,pct)=x+trunc(x*pct/100)`, `trackFactor(base,[n,d])=trunc(base*n/d)`, `trackMaxHit=trunc((effLvl*gearBonus+320)/640)`. **`trackFactor` multiplies before dividing and can exceed 2³¹ → use `long`** (`PlayerVsNPCCalc.ts:567`).
- **`floor` differs from `trunc` on negatives.** Upstream's one `Math.floor` (`PlayerVsNPCCalc.ts:479`) is on a non-negative value; port floor as `Math.floorDiv`/`Math.floor` only where upstream uses floor, and `(long)`-truncation everywhere upstream uses `trunc`.
- **No Float32/`fround`** (verified absent) — Java `double` matches the value type exactly.
- **`getHash()` uses BigInt** (`HitDist.ts:99-106`) — port with `long`/`BigInteger`. The delay pack/unpack (`HitDist.ts:254,264`) uses **signed 32-bit** semantics and a quirky `0x8F000000` mask (not `0xFF000000`) — port as Java `int` **verbatim**; it is a latent quirk to replicate, not fix.
- **`Math.pow` on non-integer bases is last-ULP divergent** across V8/JVM — where upstream raises to an *integer* power, port as **repeated multiplication**, not `Math.pow` (`claws.ts:61`).
- **Summation order matters** (§4.4). Where upstream folds a `Map`, use `LinkedHashMap` to preserve insertion order, skip NaN, and `|| 0` empty.
- **Add `strictfp`** to all ported calc classes (the JVM dropped default strictfp; FMA contraction would otherwise threaten reproducibility).

### 4.8 Sanitization / init order (`BaseCalc` constructor)

Replicate exactly (`BaseCalc.ts:83-102`): `scaleMonster(monster)` unless `disableMonsterScaling`/`noInit` (keep `baseMonster` for the spec sub-calc), canonicalize equipment, build the equipped-item name list, then `sanitizeInputs()`: default `monsterCurrentHp` to `skills.hp` when unset/over-max, set `demonbaneVulnerability` for demons, null the spell on non-cast stances or wrong weapon/monster (emit `SPELL_WRONG_WEAPON`/`SPELL_WRONG_MONSTER`), validate ammo (`EQUIPMENT_WRONG_AMMO`/`MISSING_AMMO`), emit unsupported-effect `UserIssue`s. `getSpecCalc()` rebuilds from `baseMonster` with `usingSpecialAttack=true`.

### 4.9 Performance optimizations (post-parity only)

Faithful first, refactor second — parity tests are the contract that lets us refactor safely. Then: cache per-query invariants (monster defence rolls, base prayer multipliers, monster scaling — invariant across candidate gear in one optimizer pass); a DPS-only fast path that defers full `HitDistribution`/histogram until final scoring; avoid hot-path allocation.

### 4a.3 Optimizer assumed-state policy (v0.2)

The optimizer must *choose* a player state per candidate, deterministically. Policy:

- **Prayer:** the max relevant offensive prayer the account level allows for the candidate's style — Piety (melee) / Rigour (ranged) / Augury (magic), degrading to lower variants only if level/unlock is insufficient.
- **Potion:** the best applicable potion **present in the bank** for that style, max-combined via `BoostsCalculator`.
- **`onSlayerTask`:** config toggle (default off).
- **All situational/spec-only buffs** (`soulreaperStacks`, `baAttackerLevel`, `chinchompaDistance`, `chargeSpell`, `markOfDarknessSpell`, `usingSunfireRunes`, `forinthrySurge`, `inWilderness`): **off** unless the encounter metadata or config explicitly enables them.

The chosen prayer/potion/on-task assumption **is** the "assumptions" string surfaced on the loadout card (§9).

## 6. Item & slot reconciliation (NEW — the riskiest integration point)

### 6.1 Same ID space (verified)

RuneLite `Item.getId()` and weirdgloop `equipment.json` `id` are the **same OSRS cache item-ID space**. weirdgloop's `id` is populated directly from the wiki `item_id` (`scripts/generateEquipment.py:144`), which is the cache ID RuneLite uses. Spot-verified: 20997 Twisted bow, 4151 Abyssal whip, 12926 Toxic blowpipe (Charged), 11832 Bandos chestplate, 21012 DHCB — all identical to RL `ItemID` constants. **No remapping table is needed.** `id` is effectively a primary key: of 5306 rows there is exactly **one** collision (32640, two non-equipment leagues joke items) — guard against it at load.

### 6.2 `version` is descriptive, not a lookup key (revision-1 error corrected)

Each charged/uncharged/poisoned state is its **own cache id with its own row** (Trident of the seas → 11905/11907/11908; Sanguinesti → 22323/22481; Bow of faerdhinen → 25865 Charged / 25862 Inactive). So the bank id the player owns **already selects the correct row** — there is **no `(id,version)` disambiguation step**. `version` is human-readable metadata. (This refutes revision 1's premise that lookup was "undefined"; upstream's own production lookup is `availableEquipment.find(e => e.id === id)`, `state.tsx:154`.)

### 6.3 The mapping pipeline (per bank item)

1. `rlId = itemManager.canonicalize(bankItem.getId())` (`ItemManager.java:467-482`) — resolves **only** noted→unnoted (`getLinkedNoteId`), placeholder→real (`getPlaceholderId`), and a small fixed `WORN_ITEMS` set (graceful/skillcape/hunter). It does **not** collapse degraded/recolour/locked/charge variants.
2. `canonicalId = aliases.getOrDefault(rlId, rlId)` — port weirdgloop's `getCanonicalItemId` by shipping `equipment_aliases.json` (flat `{variantId: baseId}`, 828 entries). **This** stage collapses degraded→undamaged (e.g. Ahrim's hood 4856 `'100'` → 4708 `'Undamaged'`), `(or)`/`(g)` recolours, locked, Moons, etc., and swaps in the base row's combat stats.
3. `equipmentById.get(canonicalId)` against `equipment.json`. **One row per id**; use its `bonuses/offensive/defensive`. If absent after both steps, the item has **no DPS stats — skip it** (count it for the "N unrecognized" indicator, §7.5).

### 6.4 Equipment-slot mapping

weirdgloop has **11 slot strings**; RuneLite `EquipmentInventorySlot` has 14 worn indices (incl. cosmetic ARMS/HAIR/JAW). For **bank** items there is no worn slot at all — **read `slot` from the matched `equipment.json` row**, never compute it.

| weirdgloop slot | RL worn slot (for "use my gear") | Notes |
|---|---|---|
| head | HEAD(0) | |
| cape | CAPE(1) | |
| neck | AMULET(2) | |
| weapon | WEAPON(3) | `isTwoHanded=true` ⇒ occupies weapon **and excludes shield** |
| body | BODY(4) | |
| shield | SHIELD(5) | empty when a 2H weapon is worn |
| legs | LEGS(7) | |

There is **no weirdgloop `2h` slot** — `generateEquipment.py:182-185` collapses wiki `2h` into `slot=weapon` + `isTwoHanded=true` (751 rows).

### 6.5 Failure modes & handling

| Case | Handling |
|---|---|
| Item absent from `equipment.json` after both steps (non-equipment, skipped wiki items like Wristbands of the arena canonical 26721/26723/26727, or brand-new items) | Exclude from the feed; surface as "no stats / data may be outdated", never zero stats. |
| Placeholder (qty 0, `getPlaceholderTemplateId != -1`) | Canonicalize to the real id but treat as **not owned** for availability. |
| Noted item | `canonicalize` resolves via `getLinkedNoteId` before lookup. |

| Weapon sub-state (blowpipe dart `itemVars`) | Not recoverable from a bank id; default/leave to the v0.1 panel. |
| `getName()==null` (invalid/historical id) | Skip. |

`EquipmentRepository` indexes by `id` (matching upstream) and applies the canonicalize+alias pipeline before lookup.

### 7.1 RuneLite API surface (verified against the client)

- **Containers:** `Client.getItemContainer(InventoryID)`; gameval ids **BANK=95, INV=93, WORN=94** (the `net.runelite.api.InventoryID` enum is deprecated — prefer gameval ints). `Item` exposes `int id, int quantity` only — resolve names/stats via `ItemManager`. WORN is indexed by `EquipmentInventorySlot.getSlotIdx`.
- **Stats:** `getRealSkillLevel(Skill)` (base) vs `getBoostedSkillLevel(Skill)` (current). We use **base** + computed boosts (§4a). Avoid `Skill.OVERALL`.
- **State:** `Client.isPrayerActive(Prayer)`; spellbook via `getVarbitValue(4070)`; Tome-of-fire charges via varbits 16153/16152 (examples of auto-detect sources for §4a buffs).
- **Bank freshness:** `ItemContainerChanged` for BANK is a **stale snapshot** — it reflects the *last opened* bank. Detect the bank being open via `WidgetLoaded(InterfaceID.BANKMAIN)` and surface "bank last seen at <time>" (§7.5).
- **Equipment stats note:** RuneLite *does* expose `ItemManager.getItemStats(...).getEquipment()` (`ItemEquipmentStats`), but we **do not use it** — to stay parity-equal with weirdgloop we read bonuses from the bundled `equipment.json`. (Recorded so an implementer doesn't "helpfully" swap in RL stats and break parity.)

### 7.2 Panel wiring idiom (from `GrandExchangeSearchPanel` / `bank`)

`@Provides` the `BossBisConfig`; register the panel via `injector.getInstance` + `NavigationButton` + `ClientToolbar.addNavigation`. Mutate Swing **only** on the EDT (`SwingUtilities.invokeLater`); read game state **only** on the client thread (`ClientThread.invokeLater`).

### 7.3 Threading model (NEW)

Upstream runs every calc in a Web Worker because a full calc + TTK fold is heavy; the v0.2 optimizer fans out potentially thousands of `forPlayerVsNpc` calls. Therefore:

- **Calc + optimizer run on a dedicated background `ExecutorService`** (a single-thread executor owned by the plugin, or the injected `ScheduledExecutorService`). **Never** on the client/game thread (stutters the game) and **never** on the EDT (freezes the panel).
- **Bank/equipment reads** happen on the client thread (the `ItemContainerChanged` handler already fires there) and are **immediately snapshotted** into an immutable `BankSnapshot` (an `int[]` of ids/quantities) before hand-off. The background thread never touches `Client` directly; mid-search game reads go through `ClientThread.invokeLater`.
- **UI updates** marshalled back to the EDT via `SwingUtilities.invokeLater`.
- **Cancellation:** each optimizer run is tagged with the `BankSnapshot` generation it was computed against. If the bank changes or the user selects a different encounter mid-search, the in-flight run is cancelled (interrupt/flag) and its results discarded — preventing stale loadouts from overwriting fresh ones. Precedent: `GrandExchangeSearchPanel` (`executor.execute → clientThread.invokeLater → SwingUtilities.invokeLater`).

### 7.4 Config schema (`BossBisConfig`)

| Key | Type | Default | Drives |
|---|---|---|---|
| `assumeOnSlayerTask` | boolean | false | `buffs.onSlayerTask` in the optimizer policy |
| `assumedPrayerMelee/Ranged/Magic` | enum | Piety/Rigour/Augury | optimizer prayer policy (degrades if unavailable) |
| `usePotionsFromBank` | boolean | true | whether the optimizer applies the best in-bank potion |

| `showDataVersion` | boolean | true | show the weirdgloop snapshot version in the panel |

### 7.5 Empty / error / not-logged-in states

| State | Behaviour |
|---|---|
| Not logged in | Disable autofill; allow manual entry in v0.1; v0.2 boss browser shows "log in to scan your bank". |
| Bank never opened this session | `ItemContainerChanged` hasn't fired → prompt "open your bank once so BossBiS can read it". |

### 7.6 Privacy

The plugin makes **zero network requests**; all data is bundled in the jar. The bank snapshot is held in memory and optionally persisted to the RuneLite profile (config) keyed per account; it is **never transmitted**. i18n/a11y are out of scope — English-only, following RL panel conventions (an explicit choice, not an omission).

### 7.7 Memory footprint

`equipment.json` (3.1 MiB) + `monsters.json` (2.4 MiB) parse into thousands of `@Value` POJOs; naive eager full-parse can retain 20–60 MiB live for the plugin lifetime, which matters because the hub runs many plugins in one JVM. **Strategy:** parse `equipment.json` eagerly (needed for any bank scan) but **lazy-load `monsters.json`** — index id→offset (or load only the monsters referenced by shipped encounters, ~50–100) and parse on demand. Intern shared strings. **Budget: < 15 MiB retained**, asserted by a heap test. (Note: this is *heap*, distinct from the ~5.4 MiB *jar* size in §3.4.)

## 8. Optimizer design (v0.2)

### 8.1 Encounter model (with `MonsterInputs`)

A boss is a `BossEncounter`, not a bare monster:

```java
@Value public class BossEncounter {
  String name;                       // "Vorkath"
  int primaryMonsterId;              // 8059 (Post-quest) — NOTE: 8061 (revision 1) does NOT exist in monsters.json
  String primaryMonsterVersion;      // disambiguates monsters.json rows with the same id+name
  MonsterInputs primaryInputs;       // per-encounter scaling/prayer/phase defaults (e.g. ToA invocation, party size)
  List<SecondaryTarget> secondary;   // minions/phases with time weights
  Requirements requirements;         // mandatory + recommended items
}
@Value public class SecondaryTarget {
  int monsterId; String monsterVersion;
  double timeWeight;                 // approximate planner heuristic, NOT a measured value
  StyleHint preferredStyle;          // optional switch hint
}
@Value public class Requirements { List<Requirement> mandatory, recommended; }
@Value public class Requirement {
  String category;                   // "antifire", "crumble-undead-runes", "anti-venom"
  List<Integer> oneOf;               // substitutable item ids
  EnumSet<StyleCondition> conditions;// TYPED (was Map<String,Object>): e.g. {MAGIC} ⇒ applies only to magic loadouts
}
```

`MonsterInputs` defaults come from `INITIAL_MONSTER_INPUTS` (§4.2). Curated encounters override per-encounter (a ToA entry sets `toaInvocationLevel`/`partySize`; a praying boss sets `inputs.prayers`/`phase`).

### 8.2 Monster protection-prayer policy

A boss praying against a style **zeroes that style's damage** (`PlayerVsNPCCalc.ts:2108-2118`, `isImmune()` returns true in the standard path). This is an **encounter fact**, not within-kill micro-opt. Encounters where the boss prays (or alternates phases) must set `inputs.prayers`/`phase` so the optimizer doesn't recommend a zero-damage style. Encounters that can't be modeled this way in v0.2 are excluded by name (§8.7).

### 8.3 Search algorithm (joint, corrected)

Per `BossEncounter`, optimize the **primary** loadout as an outer enumeration; greedy is used **only** where it is provably admissible:

1. **Enumerate weapons** present in the bank (typically 5–50).
2. **Enumerate attack styles / spells** valid for that weapon (each stance Accurate/Aggressive/Controlled/Rapid/Longrange as applicable; for magic weapons, each usable combat spell). Style/spell gate damage type, max hit, accuracy, and speed — they are **part of** the joint search, not fixed.
3. **Enumerate ammo / shield** consistent with the weapon (2H constraint, ammo-type compatibility).
4. **Enumerate set-templates** applicable to this (weapon, style): the empty template plus every set whose pieces exist in the bank — **Void, Crystal (+bowfa), Inquisitor, Justiciar, Obsidian, Dharok, Black-mask/slayer-helm variants, and the salve-vs-slayer-helm neck/head mutex**. A template **pins** the slots it occupies.
5. **For the slots NOT pinned by the template**, fill by **per-slot Pareto-best** (greedy) on the relevant additive bonuses. Greedy is admissible **here and only here**: with set membership fixed, each remaining slot's bonus is determined, so its marginal DPS no longer depends on other free slots' choices (the residual problem is separable — `PlayerVsNPCCalc.ts:229,378` rolls are additive once gear-multipliers are fixed). **Greedy across set-affecting slots *before* fixing the set is unsound and is not used** (this was revision 1's bug).
6. **Full-calc score** the completed candidate against the whole encounter, weighted across primary + secondary per `timeWeight`. Keep the best (single best in v0.2; top-K deferred to v0.3).

### 8.4 Complexity & budget (replacing the unsubstantiated 200 ms)

Candidate count ≈ `|weapons| × |styles+spells/weapon| × |ammo/shield| × |applicable set-templates|`; per candidate, greedy fills `O(free-slots × items/slot)` with one full calc per trial. With `|weapons|≤50, styles≤5, ammo/shield≤10, set-templates≤8`, and the DPS-only fast path in the greedy inner loop (defer full `HitDistribution` until step 6, §4.9), the dominant cost is a few **10³–10⁴ full-calc evaluations** for a 500-item bank after Pareto pruning. **Gate on a deterministic eval-count ceiling** (cap set-template enumeration to sets with ≥2 pieces in bank if exceeded) **plus** a generous warmed-JMH wall-clock ceiling on a pinned reference machine — not a single hard-coded millisecond absolute. Raid-scaled encounters (heavier per-eval) are budgeted separately.

### 8.5 Optimizer correctness testing (brute-force oracle is the primary gate)

- **Exhaustive-oracle (primary):** on small synthetic banks (≤ ~3–8 candidates/slot, with realistic 2H/ammo constraints and at least one set-bonus interaction per case), brute-force the true global-max-DPS loadout over all (weapon, style/spell, ammo, gear) combinations and assert the optimizer matches it **exactly**. A test matrix covers each set interaction (Void, Crystal, Inquisitor, Justiciar, Obsidian, Dharok, salve/slayer-helm). Any mismatch is a bug. Where greedy provably can't match for a documented case, assert a declared optimality bound (e.g. ≥99% of brute force) instead of silently passing.
- **Property (smoke, demoted):** chosen DPS ≥ best of 100 random alternates — explicitly noted as a weak floor, not a proof.
- **Regression scenarios:** tbow chosen at a magic-defence-weak boss; slayer helm + on-task ⇒ helm included; salve(e) ⇒ undead; etc.

### 8.6 Encounter JSON schema & validation

- A versioned `encounter.schema.json` under `src/main/resources/org/bossbis/data/encounters/`.
- A build-time `EncounterValidationTest` asserts: (i) every file validates against the schema; (ii) every `primaryMonsterId`+version and `SecondaryTarget` resolves to exactly one `monsters.json` entry; (iii) every `Requirement.oneOf` id resolves in `equipment.json`; (iv) `timeWeight ∈ (0,1]`. **CI fails on any unresolved id** — this would have caught revision 1's non-existent Vorkath `8061`.
- `timeWeight`s are documented as **approximate planner heuristics**, not measured values.

### 8.7 Output to UI

Per encounter card: best loadout (11-slot grid), **DPS with the assumed prayer/potion/on-task labeled** (§4a.3), required-items checklist (✓ have / ✗ missing, with `oneOf` substitutes), gap callouts ("with Bandos chestplate: +0.34 DPS"), switches for secondary targets + their inventory-slot cost, and a "data: weirdgloop@`<commit>`, `<date>`" line.

### 8.8 v0.2 scope guard

v0.2 ships **~10** curated encounters initially (expanding once the schema/validator/oracle are proven), restricted to **solo, single-phase, non-team-raid** bosses. Raid-scaled (ToA/CoX/ToB) and multi-phase-praying encounters are explicitly **deferred** — the calc *can* score them given inputs, but curating correct per-encounter `MonsterInputs` for them is v0.3+ work, and the v0.2 encounter list names its exclusions rather than implying universal coverage.

## 11. Risks & mitigations

| Risk | Mitigation |
|---|---|
| **GPLv3 not accepted on the Plugin Hub** | Automated gate accepts any redistributable LICENSE (GPLv3 qualifies); but it's undocumented/unconventional. **Confirm with maintainers before v0.1.8.** Fallback: clean-room BSD reimplementation (large, out of scope). |
| Float parity unattainable | Two-tier contract (integers exact, floats ≤1e-9), `strictfp`, reproduced d3 summation order, verbatim bit-masks; bitwise identity is a non-goal. |
| `src/lib` non-self-contained (d3/lodash/utils) | Ported `support/` package + dedicated `RuntimeSupportParityTest`; widened dev submodule scope. |
| Greedy optimizer suboptimal | Joint search with greedy only post-set-template; **brute-force oracle test** is the primary correctness gate. |
| Monster scaling / protection prayers mis-scored | `MonsterInputs` modeled end-to-end; encounter metadata sets prayers/phase; v0.2 scope explicitly excludes encounters that can't be curated correctly. |
| Wrong encounter data silently → confident-wrong BiS | `encounter.schema.json` + build-time id-resolution validator; start with 10 encounters. |
| Game stutter / Swing corruption (hub blocker) | Background executor for calc/optimizer; client-thread bank read snapshotted; UI on EDT; in-flight cancellation. |
| 5.4 MiB data → heap blowup | Eager equipment parse, lazy monster load, string interning, <15 MiB retained-heap budget + test. |
| Jar > 10 MiB after a data bump | Headroom now; if exceeded, `jarSizeLimitMiB=` (needs maintainer approval). |
| Upstream refactor mid-port | Pinned submodule; bump deliberately; corpus diff = changelog; `PORT_STATE.md` honest about gaps. |
| Bespoke corpus generator has silent bugs | **Avoided** — reuse upstream's 26 Jest tests + `TestUtils`; thin shim only for extra rows. |

## Appendix D — Glossary

**Parity corpus** — committed input/output rows (ported upstream Jest cases + `TestUtils`-generated rows + fuzz) verifying the Java calc against weirdgloop. **PORT_STATE.md** — truth table of which TS files are ported, at which commit. **BossEncounter** — curated record: primary monster + `MonsterInputs` + secondary targets + requirements. **Set-template** — a recognized armour set (Void/Crystal/Inquisitor/…) that pins its slots so the rest can be filled greedily. **Switches** — extra gear for secondary targets, costing inventory slots.
