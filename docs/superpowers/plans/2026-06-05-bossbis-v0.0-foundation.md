# BossBiS v0.0 Foundation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

> **Revision 2 (2026-06-05):** Regenerated against design spec rev 2. The original v0.0 plan assumed a `:calc`/`:plugin` two-module monorepo, JitPack publication, and a build-time copy of JSON from a git submodule. The audit established that the **RuneLite Plugin Hub builds exactly one repo into one jar**, so this plan now produces a **single hub-style repo** (modeled on `runelite/example-plugin`): the calc is vendored as internal `org.bossbis.calc.*` source, the item/monster JSON is **committed** into `src/main/resources`, the weirdgloop checkout is a **dev-only** git submodule (not a build input), and there is **no JitPack**. See spec §3.

**Goal:** Stand up the BossBiS GitHub repo as a buildable, testable, hub-shaped RuneLite plugin skeleton — Groovy Gradle build (mirroring `runelite/example-plugin`), a minimal launchable `BossBisPlugin`, the committed weirdgloop data snapshot loading from the classpath (proven by a green smoke test), a determinism (RNG) guard, the dev-only weirdgloop submodule + data-refresh script, the Node corpus-generator scaffold, `PORT_STATE.md`, GitHub Actions CI, and a `v0.0.0` tag — with **no** calc or optimizer logic yet.

**Architecture:** One public GitHub repo `bossbis`, single Gradle project (Groovy DSL, `id 'java'`, `compileOnly net.runelite:client:latest.release` from `repo.runelite.net`, Java 11 bytecode). Plugin code under `org.bossbis.plugin.*`; calc code will later be vendored under `org.bossbis.calc.*` (empty at v0.0). Item/monster JSON is committed under `src/main/resources/org/bossbis/data/` and loaded via `Class.getResourceAsStream` (the hub never unpacks the jar to disk). The weirdgloop repo lives at `tools/upstream/` as a sparse, dev-only git submodule used for porting reference and corpus generation; the production build does not touch it. Tests use JUnit 5 + AssertJ as `testImplementation` (test deps are not bundled into the plugin jar, so they don't hit the hub's runtime-dependency allowlist).

**Tech Stack:** Java 11, Gradle (Groovy DSL) with the RuneLite Maven repo, Lombok, Gson (transitive from `net.runelite:client`, **injected** — never `new Gson()` in `main`), JUnit 5 + AssertJ (test only), Node 20 + TypeScript (dev tools), GitHub Actions, GPLv3.

**Working directory:** `/home/alexb/runelite/projects/bossbis/` (created in Task 1). Do **not** run these tasks inside the existing `runelite` checkout.

**Prerequisites (verify once before starting):**
- `java -version` → 11 or higher.
- `gh auth status` → authenticated to github.com.
- `node --version` → **20.11 or higher** (the corpus scaffold uses `import.meta.dirname`, added in Node 20.11).
- `git --version` → **2.28 or higher** (`git init -b main` in Task 1 needs 2.28; this also covers the 2.25+ floor for sparse-checkout cone mode).
- A Gradle to bootstrap the wrapper: system `gradle` (`sudo apt-get install -y gradle` on WSL Ubuntu), SDKMAN (`sdk install gradle 8.7`), or copy `gradlew`/`gradlew.bat`/`gradle/wrapper/` from the existing `/home/alexb/runelite` checkout. **If you copy the wrapper**, normalize line endings and perms first: `sed -i 's/\r$//' gradlew && chmod +x gradlew` — a CRLF `gradlew` (the runelite checkout has `autocrlf` on) fails with "cannot execute: required file not found". The primary path (`gradle wrapper …` in Task 5) writes LF and avoids this.

**Conventions for the whole plan:**
- All Gradle files use **TAB indentation** and stay **≤120 columns** (the hub's `build=gradle` lint requires this).
- Commit messages use the prefixes already established for this project (`chore:`, `build:`, `test:`, `docs:`, `ci:`, `tools:`).

Create `/home/alexb/runelite/projects/bossbis/tools/update-data.sh`:

```bash
#!/usr/bin/env bash
# Refresh the committed weirdgloop data snapshot from the dev submodule.
# Usage: tools/update-data.sh [<weirdgloop-commit>]
# With an argument, checks the submodule out at that commit first.
set -euo pipefail

REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"
SUB="$REPO_ROOT/tools/upstream"
DEST="$REPO_ROOT/src/main/resources/org/bossbis/data"

if [ "${1:-}" != "" ]; then
	git -C "$SUB" fetch origin "$1"
	git -C "$SUB" checkout "$1"
fi

mkdir -p "$DEST"
for f in equipment.json monsters.json equipment_aliases.json spells.json; do
	cp "$SUB/cdn/json/$f" "$DEST/$f"
done

COMMIT="$(git -C "$SUB" rev-parse HEAD)"
DATE="$(git -C "$SUB" show -s --format=%cs HEAD)"
cat > "$DEST/data-version.properties" <<EOF
# Generated by tools/update-data.sh — do not edit by hand.
weirdgloop.commit=$COMMIT
weirdgloop.snapshotDate=$DATE
EOF

echo "Copied 4 JSON files + data-version.properties from weirdgloop@$COMMIT ($DATE)"
```

```java
package org.bossbis.calc;

import com.google.gson.JsonArray;
import com.google.gson.JsonParser;
import java.io.InputStream;
import java.io.InputStreamReader;
import java.nio.charset.StandardCharsets;
import java.util.Properties;
import org.junit.jupiter.api.Test;
import static org.assertj.core.api.Assertions.assertThat;

class BundledDataSmokeTest
{
	private InputStream open(String name)
	{
		// Mirrors how the calc will load data in production: classpath, never the filesystem.
		return getClass().getResourceAsStream("/org/bossbis/data/" + name);
	}

	private JsonArray parseArray(String name) throws Exception
	{
		try (InputStream in = open(name))
		{
			assertThat(in).as("%s must be on the classpath", name).isNotNull();
			// NOTE: net.runelite:client pins Gson 2.8.5 transitively, which has only the
			// INSTANCE method new JsonParser().parse(Reader) — the static JsonParser.parseReader
			// was added in Gson 2.8.6. The calc's production data loader (v0.1.0) must likewise
			// target the client's Gson 2.8.5 API.
			return new JsonParser().parse(
				new InputStreamReader(in, StandardCharsets.UTF_8)).getAsJsonArray();
		}
	}

	@Test
	void equipmentJsonIsBundledAndLarge() throws Exception
	{
		JsonArray arr = parseArray("equipment.json");
		assertThat(arr.size()).as("equipment rows").isGreaterThan(5000);
	}

	@Test
	void monstersJsonIsBundledAndLarge() throws Exception
	{
		JsonArray arr = parseArray("monsters.json");
		assertThat(arr.size()).as("monster rows").isGreaterThan(2000);
	}

	@Test
	void aliasesAndSpellsAreBundled() throws Exception
	{
		try (InputStream a = open("equipment_aliases.json"); InputStream s = open("spells.json"))
		{
			assertThat(a).as("equipment_aliases.json").isNotNull();
			assertThat(s).as("spells.json").isNotNull();
		}
	}

	@Test
	void dataVersionIsRecorded() throws Exception
	{
		Properties p = new Properties();
		try (InputStream in = open("data-version.properties"))
		{
			assertThat(in).as("data-version.properties").isNotNull();
			p.load(in);
		}
		assertThat(p.getProperty("weirdgloop.commit"))
			.as("recorded weirdgloop commit")
			.hasSize(40);
	}
}
```

## Task 9: Determinism (RNG) guard

The calc must contain no nondeterministic randomness (spec §4.7). A guard catches an accidental `Math.random()`/`new Random()` in `org.bossbis.calc` long before it corrupts parity.

**Files:**
- Create: `/home/alexb/runelite/projects/bossbis/tools/check-determinism.sh`

- [ ] **Step 1: Write the guard**

Create `/home/alexb/runelite/projects/bossbis/tools/check-determinism.sh`:

```bash
#!/usr/bin/env bash
# Fail if the calc package uses nondeterministic randomness (breaks parity).
set -euo pipefail
REPO_ROOT="$(cd "$(dirname "$0")/.." && pwd)"
CALC_DIR="$REPO_ROOT/src/main/java/org/bossbis/calc"
```
